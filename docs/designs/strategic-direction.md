## Vision

### Core Positioning
**中国团队 + GitLab + 自托管的最佳 AI Code Review 工具**

不与 PR-Agent (10.5K stars) 正面竞争通用市场。聚焦三个竞品无法轻易复制的维度：
1. 中文 review 质量（不是翻译，而是中文工程师习惯的表达）
2. 国产模型支持（DeepSeek、通义千问等——成本低 + 数据不出境）
3. 多模型共识（同时问 2-3 个模型，合并结果提升召回率）

### 10x Check
10x 版本是：一个中国团队 5 分钟部署的 GitLab AI Review 系统，同时用 3 个模型（1个国产 + 2个国际）审查代码，输出高质量中文 review，附带团队级 review 洞察看板。

这个产品在中国市场没有直接竞品。GitLab Duo 太贵（需要 Premium + Duo 订阅），PR-Agent 不支持中文优化，CodeRabbit 企业版需要 500+ 座。

## Implementation Details

### #1 中文 Review 质量

**已有基础（current state）：**
- `internal/reviewlang/language.go`: 默认 `zh-CN`
- `internal/llm/summary.go`: 已有中文 summary prompt
- `internal/llm/processor.go`: 系统 prompt 已支持 `output_language` 注入
- 降级消息已有中文版本

**待建设 gap：**
- Review prompt 中文术语专项调优（当前 prompt 是英文写的，模型输出中文但术语不够地道）
- Finding 级别的中文模板优化
- Severity 显示层映射
- Summary note 中文模板独立设计（非英文翻译）

**Severity 映射（显示层，不改数据库）：**
| DB value | 中文显示 |
|----------|---------|
| critical | 严重 |
| high | 警告 |
| medium | 注意 |
| low | 建议 |
| info | 信息 |
| nit | 细节 |

Severity 在数据库中保持英文存储，中文标签仅在 GitLab comment/summary 渲染时应用。

**验收标准：**
- 对 10 个样本 MR 做盲评，由 2 名中文工程师打分（自然度 1-5），目标均分 ≥ 4
- 代码术语保持英文（变量名、函数名不翻译）

### #3 轻量部署 — 分阶段实现

**CC = Claude Code 实现时间（单开发者 wall-clock）**

**Phase 1: Demo / CI pipeline 模式**（~8h CC）
- **架构选择：** 不复用现有 `Processor`（它深度耦合 `db.Queries`）。新建一个 `cmd/review-cli/main.go` 轻量管线：
  1. 从 stdin 读 unified diff（或接受 `--gitlab-mr-url` 参数直接拉取）
  2. 直接调 LLM provider（复用 `internal/llm/factory.go` + `registry.go`）
  3. 输出 JSON findings 到 stdout
  4. 可选：`--write-gitlab` flag 直接写 GitLab discussion（复用 `internal/gitlab/discussions.go`）
- **不含：** 数据库、Redis、历史记录、去重、audit log
- **适用于：** GitLab CI job / 本地命令行快速体验
- **作为 `.gitlab-ci.yml` 模板提供一键集成示例**

**Phase 2: SQLite 模式** ✅ **已完成** (2026-03-27)
- `db.Store` 接口抽象 MySQL/SQLite
- `internal/db/sqlitedb/` 手写 60+ 方法，返回共享 `db.*` 模型类型
- DSN 前缀自动检测 (`sqlite://` → SQLite)
- 所有生产调用点已切换为 store factory 模式
- Redis 已是可选的（内存限流 fallback）
- 571 测试通过

**Migration & Compatibility:** SQLite 模式是 MySQL 模式的功能子集。MySQL 部署保持为完整功能模式。两者不可互相迁移。

### #6 多模型共识 — 算法规范

**v1.0: 粗匹配共识（~8h CC）**

**问题：** 当前 `dedup.go` 的 fingerprint 是 sha256(path + line + category + description)。不同模型对同一 bug 的描述不同，hash 必然不同。

**粗匹配算法：**
1. 并行调用 2-3 个 provider（需要改造 `processor.go` 当前的单 provider 顺序调用）
2. 按以下维度匹配候选组：
   - **同文件 + 行号差异 ≤ 5 行**，或
   - **同文件 + canonical_key 匹配**（已有字段，`normalizedFinding.CanonicalKey`），或
   - **同文件 + symbol 匹配**（已有字段，`normalizedFinding.Symbol`）
3. 共识分级：
   - 2+ 模型匹配到同一组 = `consensus: true`，confidence 提升一级
   - 仅 1 个模型报告 = 保持原始 confidence
4. 合并输出：取 consensus finding 中描述最详细的版本

**v1.1: 语义去重（deferred）**
- 对粗匹配未命中但可能是同一问题的 findings，增加轻量 LLM 调用做语义比较
- Prompt skeleton: `"以下两个 code review finding 是否描述同一个问题？回答 JSON: {same: bool, reason: string}"`
- 失败回退：LLM 调用超时/失败时，默认视为"非同一问题"（保守策略）

**验收标准：** 对 10 个样本 MR 跑 3 模型共识，验证：
- 真重复合并率 ≥ 90%
- 假合并率（不同问题被错误合并）≤ 5%

**依赖：** 技术上无前置依赖（可用现有 MiniMax + OpenAI + Anthropic 测试）。实际上建议 #1（中文质量）先完成，使输出质量更一致。

**优先级说明：** 虽然多模型共识护城河评分 9/10（最高），但排在 P2 是因为：(1) 需要 v1.0 基础设施稳定；(2) 用户必须先能跑起来（P0/P1）才会关心高级功能；(3) 粗匹配版本可以快速验证思路，不必等语义去重。

## Reviewer Concerns (from adversarial review)

以下问题在两轮 review 后仍需注意：

1. **Phase 1 CLI 复用度存疑：** `processor.go` 深度耦合 `db.Queries`，新建 `cmd/review-cli` 意味着两条并行管线。长期维护成本需评估——是否值得引入 storage interface 使两条路径共享核心逻辑。
2. **粗匹配行号窗口可能不够：** 不同模型可能在定义处 vs 调用处报同一问题，行号差异可达数十行。canonical_key/symbol 回退匹配缓解了部分问题，但非万全之策。
3. **中文质量验收依赖外部评审：** "2 名中文工程师盲评"是硬依赖，单人项目可能难以组织。考虑用 LLM-as-judge（如 Claude 评分）作为自动化替代指标。
